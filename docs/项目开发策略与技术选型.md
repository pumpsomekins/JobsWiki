## 为什么"每个 income type 都做一个 HTML"是浪费

你的 `income-types.json` 已经把差异结构化了 —— 21 种类型本质上只有 **4 大模式**：

| 模式 | 代表类型 | UI 结构 |
|------|---------|---------|
| **纯输入 → 单一公式** | salary, hourly, day_rate_freelance, retainer | 几个数字输入框 + 结果卡 |
| **输入 + Toggle 分支** | piece_rate, commission_only, prize_per_event, creative_royalty, platform_revenue_share | 多一个开关，展开新字段 |
| **多段叠加（hybrid）** | base_plus_commission, wage_plus_tips, per_diem_travel, salary_plus_equity | 分组显示"工资 + 附加" |
| **特殊变体** | salary_plus_equity 的 phantom_share / TUP, government_indexed 的退休金 | 嵌套子结构 |

你只要把这 **4 种模式** 用 HTML 各做一个原型，剩下 21 - 4 = 17 种都是同一套模板换字段，HTML 重做毫无新信息。

## 推荐的做法（最少返工）

**Step 1：选 3-4 个典型，用 HTML 验证交互手感**
- `salary`（最简单，验证纯输入模板）—— 比如 Engineer 或 Teacher
- `piece_rate`（验证 hasBasePayToggle 分支）—— 你已有的 `dasher.html`
- `base_plus_commission`（验证多段叠加）—— 比如 Pharma Sales Rep
- `salary_plus_equity` + variants（验证嵌套）—— 比如 Tech Engineer / Huawei

**这 4 个原型要解决的不是"长什么样"，而是**：
- Toggle 展开/收起的动效是否符合直觉
- 结果卡是显示 4 个数（base/bonus/inflation/aftertax）还是只显示 1 个
- 移动端字段太多怎么折叠（手风琴？分步？）
- 公式 `Σ(1 + inflationRate)^year` 的可视化要不要曲线图

**Step 2：原型敲定后直接进 React，不再补 HTML**

剩下 17 种类型在 React 里就是新增一份 JSON 配置 + 复用模板，每个职业 5 分钟搞定。如果继续做 HTML，你会发现：

- 改一次配色要改 21 个文件
- `getTaxRate(country)` 这种共用逻辑被复制 21 次
- 真到了 React 阶段，2400 行 HTML × 21 = 5 万行要重写

**不要拆 index.html + css + js 文件夹**。对你这个项目阶段，拆出去反而是负担。理由：

- 你只有 4 个原型 HTML，每个 2000 行内，单文件丢给 AI 修改最快。
- 拆完后 css/js 一改就要在多个文件间跳，调试时浏览器 DevTools 也更绕。
- 真正要拆是在转 React 时，**不是在 HTML 阶段**。

**HTML 阶段推荐结构**：

```
JobsWiki/
├── index.html              ← Landing page（职业目录 + 收入类型导航）
├── prototypes/             ← 4 个原型 HTML
│   ├── dasher.html             (piece_rate 模式)
│   ├── engineer.html           (salary 模式)
│   ├── pharma-sales.html       (base_plus_commission 模式)
│   └── tech-equity.html        (salary_plus_equity + variants 模式)
├── jobs/
│   ├── income-types.json   ← 收入类型注册表（你已有）
│   └── *.json              ← 每个职业一份配置
├── shared/                 ← 共用资源（HTML 阶段就一份足够）
│   ├── tax-rates.json      ← getTaxRate(country) 的数据源

    ├── images/             ← 图片统一放这里
```

**图片要不要单独文件夹**：要。理由是 CDN 优化、SEO 图片描述、批量替换、未来转 React 时只需把 `assets/images/` 整个搬到 `public/images/`，路径不变。

## 四、关键架构决策

### 1. "一个整合模板 + JSON 切换"是对的
不是"每个职业一个组件"，而是：

```tsx
// app/[country]/[occupation]/page.tsx
import occupation from '@/data/occupations/dasher.json';
import incomeTypes from '@/data/income-types.json';

export default function Page() {
  const type = incomeTypes[occupation.incomeType]; // "piece_rate"
  return <CalculatorRenderer incomeType={type} occupation={occupation} />;
}
```

`CalculatorRenderer` 内部根据 `type.code` 派发到 4 个模板之一。新增职业 = 新加一份 JSON，**不写一行 TSX**。

### 2. 公式怎么处理
你 JSON 里 `ltv_base: "ratePerTask × tasksPerHour × ..."` 是字符串。两种处理方式：

- **方案 A（推荐）**：把字符串当文档，真正的公式写在 `lib/calc/piece_rate.ts` 里的 TS 函数。类型安全、易测、性能好。
- **方案 B**：用 `mathjs` 或自写表达式解析器在运行时 eval。灵活但慢，且失去类型检查。

JSON 里的字符串保留，作为"公式说明"展示给用户（"我们怎么算的"），这本身就是 SEO 内容。

### 3. 多语言 / 多国家
路由用 `/[country]/[occupation]` 而不是 query string，每个国家版本都是独立 URL，Google 单独收录。`country` 影响：
- 税率（`tax-rates.json`）
- 货币显示
- 默认值（DoorDash 在美国 vs 加拿大费率不同）

### 4. SEO 是关键 → 选 Next.js 而不是 Vite
你这个站靠 Google 搜"how much does a DoorDash driver make"导流。必须：
- **SSG/SSR**：用户和 Google 拿到完整 HTML，不是空壳 + JS。
- **每个职业独立路由**：`/us/dasher`, `/us/engineer`...
- **元数据动态生成**：每页独立 title / description / OG image。

Vite SPA 做不到这点（要额外配预渲染），Next.js 开箱即用。

### 5. 图片为什么必须独立文件夹
- **Next.js `<Image>` 组件**会自动生成 WebP/AVIF 多尺寸，前提是文件在 `public/`。
- **SEO**：`assets/images/dasher-cover.jpg` 比 `inline base64` 能被 Google 图片搜索收录。
- **CDN 缓存**：图片走单独域名/路径，缓存策略和 HTML 分开。
- **替换效率**：换 logo 改一个文件，全站生效。

## 五、迁移路径（HTML → React）

1. **保留**：`income-types.json`、`assets/`、所有 JSON 配置 —— 零修改搬进 `public/` 和 `data/`。
2. **抽取**：`dasher.html` 第 2409 行那种 `getTaxRate(country)` 全部移到 `lib/calc/tax.ts`，写单测。
3. **重写**：HTML 里的 UI 部分用 4 个 React 模板替换。
4. **新增**：路由、SSG 配置、SEO 元数据、`<Image>` 优化。
5. **废弃**：4 个 `prototypes/*.html` 留作历史参考，不再维护。

**一句话总结**：HTML 阶段单文件 + 资源文件夹，转 React 后 1 个 Renderer + 4 个 Template + N 份 JSON，图片必须独立目录，路由按 `/[country]/[occupation]` 设计，框架选 Next.js 而不是 Vite。
